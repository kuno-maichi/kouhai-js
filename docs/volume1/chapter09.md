# 第9話：「実証実験」

「今日は心理学部の実験室をお借りしました」

　田中先生が設備の説明をしてくれている。アイトラッカー、心拍測定器、脳波計まで並んでいる。

「こんな本格的な実験をするんですか？」

　私は驚いた。単にアンケートを取るだけだと思っていた。

「エラーメッセージへの反応を、多角的に測定します」

　雅史先輩が資料を見せてくれた。

「今回は『認知負荷』の観点から分析してみたい」

「認知負荷？」

「人間の情報処理能力には限界があります」

　田中先生が説明してくれた。

「特に、ワーキングメモリは7±2個の情報しか同時に保持できない」

「つまり、エラーメッセージが複雑すぎると、理解できないということですか？」

「その通りです。さらに興味深いのは、恐怖などの感情が認知負荷を増大させること」

　これは重要な視点だ。怖いと、それだけで頭が働かなくなる。

「従来のエラーメッセージは、認知負荷の観点から設計されていません」

　雅史先輩が従来のエラーを表示した。

```
Traceback (most recent call last):
  File "test.py", line 3, in <module>
    print(hello)
NameError: name 'hello' is not defined
```

「情報量を数えてみましょう」

　田中先生がホワイトボードに書き出した。

1. Traceback（概念）
2. most recent call last（時系列情報）
3. File "test.py"（ファイル名）
4. line 3（行番号）
5. in <module>（スコープ）
6. print(hello)（問題のコード）
7. NameError（エラー種別）
8. name 'hello' is not defined（詳細）

「8個の情報要素。ワーキングメモリの限界を超えています」

「しかも、優先順位が不明確」

　私が付け加えた。

「初心者には、どの情報が重要かわからない」


◇◇◇


「それでは、実験を開始しましょう」

　最初の被験者は、22歳の大学生。プログラミング経験は半年。

「まず、従来のエラーメッセージを見せます」

　被験者の目の動きをアイトラッカーで追跡しながら、心拍数も測定している。

　画面に従来のエラーが表示された瞬間、心拍数が急上昇した。

「視線が『NameError』で止まっています」

　田中先生が指摘した。確かに、他の情報をほとんど見ていない。

「恐怖による注意の狭窄ですね」

「注意の狭窄？」

「強い感情状態では、注意が特定の刺激に集中してしまう現象です」

　つまり、「NameError」という恐ろしい単語だけに注意が向いて、解決に必要な他の情報を見落としてしまう。

「では、改良版を試してみましょう」

　私たちの新しいメッセージが表示された。

```
💡 発見：変数の定義について学べるポイントです

'hello' という名前を使おうとしましたが、まだ定義されていません。
解決方法：hello = "何か" のように、使う前に定義してください

詳細情報：test.py の3行目
```

　今度は、心拍数の上昇が穏やかだった。

「視線の動きも滑らかですね」

　被験者は順序立てて情報を読んでいる。

「情報の構造化が効果的のようです」

　雅史先輩が満足そうに頷いた。


◇◇◇


「10人目の被験者です」

　これまでの結果をまとめながら、実験を続けた。

「面白い傾向が見えてきました」

　田中先生がデータを示してくれた。

「従来版では、『概念理解』と『感情制御』に認知リソースが分散していました」

「分散？」

「エラーの意味を理解すること」と「恐怖を抑えること」に、脳の処理能力が分かれてしまう」

　グラフを見ると確かに、従来版では脳波のガンマ波（集中状態を示す）が分散している。

「改良版では、感情的負荷が軽減されているため、理解に集中できています」

「つまり、怖くない分、考える余裕ができる」

「そういうことです。これは『認知容量モデル』で説明できます」

　田中先生が新しい図を描いた。

```
総認知容量 = 課題処理 + 感情処理 + その他

従来版：課題30% + 感情50% + その他20% = 課題に集中できない
改良版：課題70% + 感情10% + その他20% = 課題に集中可能
```

「素晴らしい！」

　雅史先輩が興奮していた。

「これは論文に書ける成果だ」

　でも、私は別のことが気になった。

「先生、個人差はどうでしょう？」

「良い視点ですね」

　田中先生がデータを整理してくれた。

「確かに、効果に個人差があります。特に『認知スタイル』によって」

「認知スタイル？」

「情報処理の個人的な好み。『分析型』と『直感型』に大別されます」


◇◇◇


「分析型の人は、詳細な情報を求めます」

　田中先生が具体例を示してくれた。

「『なぜこのエラーが起きたのか』『どういう仕組みなのか』を知りたがる」

「逆に直感型は？」

「『とにかく解決したい』。詳しい説明より、具体的な手順を求める」

　これは重要な発見だ。同じ「優しいメッセージ」でも、人によって欲しい情報が違う。

「実験データを見ると...」

　雅史先輩が分析した。

「分析型の被験者は、改良版でも『詳細情報』の部分を重視している」

「直感型は『解決方法』の部分を重視」

　視線追跡データが、それを裏付けていた。

「つまり、認知スタイルに応じてメッセージを調整すべき」

「でも、どうやって判別するんですか？」

　私の質問に、田中先生が考え込んだ。

「事前の質問紙でも可能ですが...より簡単な方法もあります」

「どんな？」

「最初のエラーメッセージへの反応を見るんです」

　面白いアイデアだ。

「詳細情報をクリックする人は分析型。解決方法を先に見る人は直感型」

「リアルタイムで認知スタイルを推定して、後続のメッセージを調整する」

　雅史先輩の目が輝いた。

「技術的には実装可能です」


◇◇◇


「でも、もう一つ重要な発見があります」

　田中先生が最後のデータを見せてくれた。

「『メタ認知』の効果です」

「メタ認知？」

「自分の理解状態を理解すること。『今、自分はどれくらい理解できているか』を把握する能力」

　実験では、被験者に「どれくらい理解できたか」を5段階で自己評価してもらっていた。

「興味深いことに、改良版では自己評価の精度が向上しています」

「どういうことですか？」

「『理解できている』と答えた人は実際に問題を解決でき、『理解できていない』と答えた人は実際に苦戦している」

　つまり、自分の理解度を正確に把握できている。

「従来版では、実際は理解していないのに『理解できた』と誤認する人が多かった」

「錯覚ですね」

　雅史先輩が頷いた。

「恐怖が判断を曇らせていた可能性があります」

「メタ認知の向上は、自律的な学習につながります」

　田中先生が重要な指摘をした。

「自分の理解度を正確に把握できれば、適切な学習行動を選択できる」

　これは想定外の成果だった。単にエラーメッセージを改善するだけでなく、学習能力そのものを向上させている。

「先生、この結果は学会で発表できるレベルですか？」

　私の質問に、田中先生が満面の笑みで答えた。

「十分以上です。認知心理学とコンピュータサイエンスの学際研究として、非常に価値がある」

　実験室を片付けながら、私は達成感を感じていた。

　私たちの直感的なアプローチが、科学的に裏付けられた。

　そして、想像以上の効果があることもわかった。

　技術と心理学の融合は、確実に新しい価値を生み出している。
