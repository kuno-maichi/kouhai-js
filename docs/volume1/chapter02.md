# 第2話：「恐怖の正体」（優美視点）

エラーメッセージを見た瞬間、なぜか涙が出そうになった。

　昨日、雅史先輩に研究室に連れてきてもらってから、私はずっとこの感覚について考えている。

「おはよう、優美。今日も来てくれたんだ」

　研究室のドアを開けると、先輩が振り返った。もう朝の9時から研究してるなんて、さすが大学院生。

「はい。この『恐怖』について、もっと詳しく説明したくて」

「助かる。実は昨夜、関連論文を調べてみたんだけど...」

　先輩がモニターを見せてくれた。画面には英語の論文がずらり。

「エラーメッセージの理解度、修正効率、学習曲線...でも、感情的反応、特に『恐怖』に焦点を当てた研究はほとんどない」

「やっぱり」

　私は自分のノートパソコンを開いた。

「昨夜、私も考えてみたんです。なぜあの赤い文字が怖いのか」

「聞かせて」

　先輩が身を乗り出した。研究者モードの真剣な表情。でも、どこか優しい。


◇◇◇


「まず、視覚的なインパクトから」

　私はPythonのエラー画面を表示した。あの忌まわしい赤い文字。

```
Traceback (most recent call last):
  File "novel_manager.py", line 3, in <module>
    print(character_name)
NameError: name 'character_name' is not defined
```

「この赤い文字、何を連想しますか？」

「警告？危険？」

「私には『叱責』に見えます」

　先輩が眉をひそめた。

「叱責...」

「小学校の頃、テストで間違えた箇所に赤ペンでバツされた記憶、ありませんか？」

「ああ...」

「赤は『間違い』『失敗』『ダメ』の色なんです。少なくとも、多くの人にとって」

　私は続けた。

「次に、『ERROR』という言葉」

「エラーは単に『誤り』という意味だけど」

「でも響きが怖い。『ERROR』って、まるで裁判で有罪判決を受けたみたい」

　先輩が苦笑した。

「そんな大げさな」

「大げさじゃないです！」

　つい声が大きくなってしまった。

「小説で読者を失う瞬間って、まさにこういう時なんです。理解できない言葉が出てきて、自分が無知だと突きつけられる瞬間」

「なるほど...」

「しかも見てください、この英語の羅列」

　私は画面を指差した。

「Traceback? most recent call last? これ、初心者に分かると思います？」

「確かに専門用語が多いけど...」

「先輩、これはまるで呪文です。理解できない言葉の連続は、人を無力にします」


◇◇◇


「優美の分析、とても興味深い」

　先輩はメモを取りながら言った。

「でも、これだけじゃない気がする。Scratchの時との違いは？」

　私は二つの画面を並べた。ScratchとPython。

「Scratchでエラーに相当するのは...」

「ブロックが繋がらない時かな」

　先輩が画面を操作して見せてくれた。

「ほら、型が合わないブロックは物理的に繋がらない」

「そう！それです！」

　私は興奮して立ち上がった。

「Scratchは『できない』ことを優しく教えてくれる。でもPythonは...」

「事後に叱る」

「その通り！」

　これだ。この違いが恐怖の核心。

「Scratchは転ぶ前に手を差し伸べてくれる親みたい。でもPythonは転んだ後に『なぜ前を見てなかった』と叱る先生」

　先輩が考え込んでいる。

「つまり、予防と事後対応の違い...」

「読者心理で言えば」

　私は自分の得意分野に話を持っていった。

「読者が離脱する前に手を打つか、離脱してから理由を説明するか。当然、前者の方が効果的です」

「でも、テキスト言語で予防的エラーは技術的に...」

「難しい、ですか？」

　先輩は頷いた。

「自由度が高い分、事前にすべてのエラーを防ぐのは不可能に近い」


◇◇◇


「でも先輩」

　私は深呼吸してから続けた。

「本当の恐怖は、もっと深いところにあると思うんです」

「深いところ？」

「自己否定感です」

　先輩が真剣な顔で聞いている。

「エラーが出た瞬間、私は何を感じたと思います？」

「...分からない」

「『ああ、やっぱり私にはプログラミングなんて無理だった』って」

　その時の感覚を思い出すだけで、胸が苦しくなる。

「Scratchでは自信があったのに？」

「だからこそです。Scratchで『できる』と思った自分が、幻想だったと突きつけられた気がして」

　私は自分の手を見つめた。

「小説でも同じです。最初は楽しく書いていた人が、ある日突然『こんなの誰も読まない』と言われたら...」

「書くのをやめる」

「そう。エラーメッセージは、私にとって『お前にはプログラミングの才能がない』という宣告に聞こえたんです」

　研究室が静かになった。

　先輩が口を開くまで、少し時間がかかった。

「優美、君の感じた恐怖...それは多くの人が感じているはずなのに、誰も言語化してこなかった」

「本当ですか？」

「研究者は『理解できない』ことは問題にしても、『怖い』という感情は...」

　先輩は自嘲的に笑った。

「僕たちは、すでにエラーに慣れきってしまっているから」


◇◇◇


「先輩、質問があります」

「なんだい？」

「先輩は初めてプログラミングした時、エラーが怖くなかったんですか？」

　先輩は少し考えてから、

「正直、覚えてない。もう10年以上前だし」

「でも、きっと怖かったはず」

「なぜそう思う？」

「だって、今の先輩はエラーを『学習の機会』だと言います。それは、恐怖を乗り越えた人の言葉です」

　先輩が驚いたような顔をした。

「乗り越えた人は、乗り越える前の恐怖を忘れがち。小説でもそうです。プロの作家は『批判は成長の糧』と言いますが、最初の批判がどれだけ心に刺さったか...」

「忘れている」

「そうです。だから初心者の気持ちが分からなくなる」

　私は先輩の目を見つめた。

「でも先輩は違います。私の恐怖を、真剣に聞いてくれている」

「それは...君の視点が新鮮だから」

「違います」

　私は首を振った。

「先輩は、心のどこかで覚えているんです。最初の恐怖を。だから私の言葉に反応した」

　先輩が黙り込んだ。

　しばらくして、ぽつりと言った。

「実は...一つだけ覚えていることがある」

「なんですか？」

「中学の時、初めてのプログラミング。エラーが出て、泣きそうになった」

　意外な告白だった。

「でも、先生に『慣れれば大丈夫』と言われて、必死に慣れようとした」

「それで？」

「慣れた。でも...何人もの同級生が諦めていくのを見た」

　先輩の表情が曇った。

「僕は乗り越えられた。でも、乗り越えられなかった人たちのことを、どこかで『努力不足』だと思っていたかもしれない」

「先輩...」

「優美、君のおかげで思い出した。恐怖は本物だ。そして、それを無視してきた僕たちの方が間違っていた」


◇◇◇


「じゃあ、どうすればいいんでしょう？」

　私は前向きに聞いた。問題を特定できたなら、解決策もあるはず。

「まず、恐怖を認めることから」

　先輩がホワイトボードに書き始めた。

『エラーに対する恐怖の要素』
1. 視覚的威圧感（赤い文字）
2. 理解不能な専門用語
3. 事後的な叱責感
4. 自己否定感の増幅

「これを一つずつ解決していけば...」

「でも先輩、技術的に可能なんですか？」

「分からない。でも...」

　先輩が振り返った。その目には、研究者の情熱が宿っていた。

「やってみる価値はある。君の言う通り、これは技術の問題じゃない。人の心の問題だ」

「はい！」

　私は力強く頷いた。

「私の恐怖を、誰かの希望に変えられるなら」

「必ず変えよう。ビジュアル言語とテキスト言語の間に、新しい道を作ろう」

　その瞬間、私たちの本当の研究が始まった気がした。

　恐怖を理解し、それを乗り越える方法を見つける。それは、プログラミング教育の新しい地平を開く挑戦。

　私にしか分からない恐怖が、誰かを救う鍵になるかもしれない。

　そう思うと、昨日まで感じていた恐怖が、少しだけ意味のあるものに思えてきた。
